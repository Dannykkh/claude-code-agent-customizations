# Suppressions — 플래그하지 않을 패턴

리뷰 시 아래 패턴은 발견으로 보고하지 않습니다.
오탐(false positive)을 줄여 리뷰 신뢰도를 높이기 위한 목록입니다.

---

## 일반

1. **무해한 중복** — "X가 Y와 중복"이지만 가독성에 도움되는 경우 (예: `present?`가 `length > 20`과 중복)
2. **임계값 주석 요구** — "이 임계값/상수의 근거를 주석으로 설명하라" → 임계값은 튜닝 중 자주 변경, 주석 부패
3. **충분한 어설션** — "이 어설션을 더 타이트하게" → 이미 동작을 커버하면 충분
4. **일관성만을 위한 변경** — 다른 상수의 가드 방식에 맞추려고 값을 조건 래핑
5. **실용적 정규식** — "이 정규식이 엣지 케이스 X를 처리 못함" → 입력이 제한적이고 X가 실제로 발생하지 않을 때
6. **다중 가드 테스트** — "테스트가 여러 가드를 동시에 실행" → 괜찮음, 각 가드를 개별 격리할 필요 없음
7. **무해한 no-op** — 배열에 없는 요소에 대한 `.reject` 등
8. **diff에서 이미 해결** — 리뷰 중인 diff에서 이미 수정된 사항

## 코드 스타일

9. **들여쓰기/포매팅** — 프로젝트에 linter/formatter가 설정된 경우 스타일 지적 억제
10. **import 순서** — 자동 정렬 도구(isort, eslint-plugin-import)가 있으면 무시
11. **trailing comma** — 프로젝트 규칙 따름, 별도 지적 불필요

## 프레임워크 특수

12. **ORM 관례** — 프레임워크 관례를 따르는 패턴 (예: Django의 Meta class, Rails의 concerns)
13. **설정 파일 구조** — 프레임워크가 강제하는 구조 (next.config.js, vite.config.ts)

## 테스트

14. **테스트 코드 DRY** — 테스트 가독성을 위해 의도적으로 반복하는 설정/어설션
15. **테스트 매직 넘버** — 테스트 데이터의 하드코딩된 값 (42, "test@example.com")

## 보안

16. **개발 환경 전용** — development/test 환경에서만 사용되는 시크릿/설정
17. **내부 도구** — 팀 외부에 배포되지 않는 내부 스크립트의 경미한 보안 지적

---

## 사용법

리뷰 결과에서 위 패턴과 일치하는 발견은 **자동 억제**합니다.
억제된 항목은 보고서에 포함하지 않습니다.
